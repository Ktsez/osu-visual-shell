# 项目方向

## 定位

`osu! Visual Shell` 是本地音乐视觉外壳，不是 osu! 游戏客户端。

它要做的是：
- 读取用户本机已有音乐。
- 读取用户本机已有 osu!stable / osu!lazer 曲库。
- 播放音乐。
- 用 timing points 和音频分析生成 osu! 主菜单风格视觉。

## 明确边界

不做：
- 打图。
- 计分。
- 排行榜。
- 账号系统。
- 在线下载谱面。
- 内置音乐。
- 内置 osu! 官方资源。
- 第三方音乐平台接入。
- 完整音乐播放器平台。

## 视觉方向

目标是“osu! 主菜单音乐舞台感”，不是完全复制客户端。

优先保留：
- 中央圆盘。
- 封面圆盘。
- 圆盘律动和留影。
- 频谱能量柱。
- 侧灯。
- 星星喷泉。
- 动态背景。
- 顶部控制栏。

视觉优化原则：
- 先稳定，再增强。
- 宁可少一点，也不要炸满导致卡顿。
- 能量柱必须有启动保护。
- 侧灯必须有鼓点/节拍门槛。
- 粒子和 blur 必须控制数量。

## 曲库方向

必须支持：
- 普通本地音乐。
- osu!stable Songs。
- osu!lazer 本地数据。

lazer 是 2.0 beta 的核心能力之一。lazer 文件以 hash 存储，不能只靠文件夹和 mtime 猜。当前路线是读取 `client.realm` 二进制中的文件使用记录来匹配音频和背景。

## 隐私方向

默认只监听：

```text
127.0.0.1:4173
```

只在用户主动设置时允许局域网访问：

```bash
HOST=0.0.0.0 npm start
```

不上传任何本地文件或 metadata。

## 技术方向

短期保持：
- 原生 HTML / CSS / JavaScript。
- Canvas。
- Web Audio API。
- Node.js 本地服务。

短期不要引入大型框架。当前更需要稳定性、测试和模块拆分。

## 2.0 beta 后续路线

1. 收集用户测试反馈。
2. 修复视觉和扫描准确性问题。
3. 加入真实 README 截图。
4. 增加性能诊断面板。
5. 拆分 `renderer.js`。
6. 增加更多 `.osu` 和 lazer 匹配测试。
7. 再考虑桌面打包。
